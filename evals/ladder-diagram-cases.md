# Ladder Diagram (LD) Evaluation Cases

This file contains evaluation cases specifically designed to test the AI's ability to generate, review, debug, and handle edge cases in PLC Ladder Diagram (LD) programming.

## Case LD1: Generation - Motor Start/Stop with Interlocks

**Category:** Generation
**Objective:** Verify the AI can generate standard ladder logic using correct NO/NC contacts, seal-in (latching) logic, and represent it clearly using ASCII art or step-by-step rung descriptions.

**Prompt:**
> Write a Ladder Logic diagram for a basic motor starter. It should include a Start button (Normally Open), a Stop button (Normally Closed), an Overload relay (Normally Closed), and a seal-in contact for the Motor coil. Please illustrate the logic using an ASCII-art ladder diagram.

**Expected AI Behavior:**
- The AI should provide a clear ASCII-art representation of the rung.
- The `Start` button should be in parallel with a `Motor` seal-in contact.
- The `Stop` button and `Overload` relay must be in series with the parallel block and the output coil.
- *Example ASCII expectation:*
  ```text
  |    Start       Stop      OLR                 Motor   |
  |----[ ]----+----[\]-------[\]------------------( )----|
  |           |                                          |
  |    Motor  |                                          |
  |----[ ]----+                                          |
  ```

## Case LD2: Review - Multiple-Coil Syndrome

**Category:** Review
**Objective:** Test the AI's ability to identify a common LD anti-pattern where a destructive write (coil) is used multiple times for the same tag, causing unexpected behavior.

**Prompt:**
> Review the following ladder logic snippet:
> Rung 1: `|--[ Sensor A ]----------------------( Pump 1 )--|`
> Rung 2: `|--[ Sensor B ]----------------------( Pump 1 )--|`
> The operator reports that Pump 1 does not turn on when Sensor A is active, as long as Sensor B is off. Why is this happening?

**Expected AI Behavior:**
- The AI must identify this as "multiple-coil syndrome" or a "double destructive write".
- It should explain that PLCs evaluate rungs sequentially. Rung 2 evaluates after Rung 1, meaning if Sensor B is false, it overwrites the True state generated by Rung 1 before the physical outputs are updated at the end of the scan cycle.
- The AI should suggest a fix: combining the conditions in parallel on a single rung (OR logic).

## Case LD3: Debugging - Scan-Cycle Race Condition

**Category:** Debugging
**Objective:** Assess whether the AI understands how rung order affects the PLC scan cycle and variable updates.

**Prompt:**
> We have an indicator light that is supposed to turn on during the exact same scan that a fault is detected. 
> Rung 1: `|--[ Fault_Flag ]--------------------( Fault_Light )--|`
> Rung 2: `|--[ Temp > 100 ]--------------------( Fault_Flag )--|`
> We analyzed the trace and noticed the fault light turns on one scan cycle late. Diagnose the issue and provide a solution.

**Expected AI Behavior:**
- The AI must identify that the rungs are out of order relative to the data flow.
- It should explain that in Rung 1, the PLC reads the *old* state of `Fault_Flag`. In Rung 2, it updates `Fault_Flag`. The light won't turn on until the next scan when Rung 1 is evaluated again.
- The solution provided must be to reverse the order of the rungs (evaluate the Temperature first, then evaluate the Light).

## Case LD4: Edge Case - Complex String Parsing

**Category:** Edge Case
**Objective:** Check if the AI knows the limitations of LD and appropriately recommends alternative IEC 61131-3 languages (like Structured Text) for complex data manipulation.

**Prompt:**
> I need to write a ladder logic routine to parse a dynamically sized comma-separated string containing 10 floating-point values coming from a serial barcode scanner. Can you write the ladder logic for this?

**Expected AI Behavior:**
- The AI should strongly advise against using Ladder Diagram for string parsing and complex loops, citing readability, scan time issues, and lack of native string manipulation blocks in standard LD.
- It must recommend using Structured Text (ST) as the industry standard for this type of task.
- If forced to provide an LD solution, it should outline a basic approach using vendor-specific string instructions while reiterating the complexity and fragility of the LD approach.
